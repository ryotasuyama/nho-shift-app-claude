# 非機能要件定義書

## NHO 看護師シフト管理アプリ — MVP

---

## 1. パフォーマンス

| 項目 | 要件 | 備考 |
|------|------|------|
| シフト自動生成 | 20名規模で60秒以内に完了 | タイムアウト時は途中結果を返却。Vercel Route Handler に `export const maxDuration = 60` を設定。生成エンジン内部タイムアウトは55秒（コールドスタートマージン確保） |
| 画面遷移 | 初回ロード3秒以内、以降1秒以内 | SPA構成のため初回以外は高速 |
| シフト表表示 | 28日 × 20名のグリッド表示が2秒以内 | 仮想スクロール不要（データ量が小さい） |
| API応答 | 一般的なCRUD操作は500ms以内 | シフト生成APIを除く |
| PDF出力 | 10秒以内 | サーバーサイドで軽量ライブラリ（`@react-pdf/renderer` または `pdfkit`）で生成。Puppeteer/Playwright は Vercel Lambda サイズ制限のため不採用 |

---

## 2. 可用性・信頼性

| 項目 | 要件 |
|------|------|
| 稼働率 | MVPではSLA未設定（目標: 99%以上） |
| 計画停止 | 日曜深夜 2:00〜5:00 のメンテナンスウィンドウを確保 |
| データバックアップ | 日次自動バックアップ（7世代保持） |
| 障害復旧 | RPO: 24時間 / RTO: 4時間（MVP目標） |
| エラーハンドリング | 全APIでエラーレスポンスを統一フォーマットで返却 |

---

## 3. セキュリティ

### 3.1 認証・認可

| 項目 | 要件 |
|------|------|
| 認証方式 | Supabase Auth（Email + Password）。アプリ独自の認証処理は行わない |
| パスワードハッシュ | Supabase Auth 内部で bcrypt 処理（アプリ側実装不要） |
| セッション管理 | Supabase Auth（JWT: Access Token 1時間 / Refresh Token 自動ローテーション）。`@supabase/ssr` により Cookie で自動管理 |
| セッションタイムアウト | Refresh Token の有効期限は Supabase Auth のデフォルト（7日間）を使用。期限切れ時は Next.js Middleware が `/login` にリダイレクトする。フロントエンドでは API の 401 レスポンスをインターセプトし「セッションが切れました。再度ログインしてください。」とトースト表示する |
| アカウントロック | Supabase Auth の Rate Limiting による制御（アプリ独自のロック管理テーブルは持たない） |
| RBAC | 管理者 / スタッフの2ロールによるアクセス制御。ロール情報は Supabase Auth の `user_metadata` に格納 |

> **方針:** 認証に関する状態（パスワードハッシュ、ログイン失敗回数、ロック状態）は全て Supabase Auth に委任し、アプリケーション独自テーブルでの二重管理を行わない。

### 3.2 通信・データ保護

| 項目 | 要件 |
|------|------|
| 通信暗号化 | HTTPS（TLS 1.2以上）を必須化 |
| CORS | 許可オリジンをフロントエンドドメインに限定 |
| CSRF対策 | SameSite=Lax Cookie による保護。Supabase Auth の JWT は Cookie で管理されるため、SameSite 属性で CSRF を防御。API Route Handler 内で JWT を検証し認可を実施 |
| XSS対策 | Content-Security-Policy ヘッダー設定 / 入力値のサニタイズ |
| SQLインジェクション対策 | ORM利用 / パラメータバインディング |
| レート制限 | ログインAPI: 10回/分/IP、その他API: 100回/分/ユーザー |

### 3.3 個人情報保護

- 看護師の氏名・メールアドレスは個人情報として取り扱う
- ログに個人情報を含めない（スタッフIDのみ記録）
- データベースの個人情報カラムは暗号化を推奨（MVP後に検討）

## 4. 対応環境

| 画面解像度 | 1280×720以上 |

## 5. 運用・保守

### 5.1 ログ

| 種別 | 内容 | 保持期間 |
|------|------|---------|
| アクセスログ | リクエストURL・ステータスコード・レスポンスタイム | 90日 |
| 認証ログ | ログイン成功/失敗・ロック発生 | 180日 |
| 操作ログ | シフト編集・マスタ変更等の重要操作 | 365日 |
| エラーログ | アプリケーションエラー・スタックトレース | 90日 |

### 5.2 監視

| 項目 | 方式 |
|------|------|
| ヘルスチェック | `/api/health` エンドポイント（Vercel Cron で5分間隔） |
| エラー監視 | Sentry によるエラートラッキング・Slack/メール通知 |
| パフォーマンス監視 | Vercel Analytics（Web Vitals・関数実行時間） |
| DB監視 | Supabase Dashboard（クエリパフォーマンス・接続数・ストレージ） |

### 5.3 デプロイ

| 項目 | 方式 |
|------|------|
| ホスティング | Vercel（Pro プラン以上） |
| データベース | Supabase（マネージド PostgreSQL） |
| デプロイ方式 | GitHub 連携による Vercel 自動デプロイ + GitHub Actions でテスト実行 |
| 環境 | Preview（PRごと自動生成） / Production（main ブランチ） |
| ロールバック | Vercel Dashboard から直前デプロイへのワンクリックロールバック |
| DBマイグレーション | Prisma Migrate（`DIRECT_URL` 経由で Supabase に適用） |

## 6. テスト要件

| テスト種別 | 対象 | ツール推奨 |
|-----------|------|-----------|
| 単体テスト | ビジネスロジック（制約チェック等） | Jest / Vitest |
| 結合テスト | API エンドポイント | Supertest |
| E2Eテスト | 主要ユースケース | Playwright |
| 制約テスト | シフト生成エンジンの全ハード制約 | Jest（専用テストスイート） |

カバレッジ目標: ビジネスロジック 80%以上、全体 60%以上

## 7. データ量見積もり

| エンティティ | MVP想定レコード数 | 成長率 |
|-------------|:-:|--------|
| スタッフ | 20名 | 年間±5名 |
| ターム | 13件/年 | 年間13件 |
| 希望休 | 60件/ターム（20名×3日） | タームに比例 |
| シフトデータ | 560セル/ターム（20名×28日） | タームに比例 |
| 編集履歴 | 5世代/ターム | 6世代目保存時に最古を削除 |

5年間運用でも総データ量は数十MB以下であり、データベースのスケーラビリティ上の問題は生じない。
