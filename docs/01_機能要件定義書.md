## NHO 看護師シフト管理アプリ — MVP

## 1. 認証・認可

### 1.1 ログイン

| 項目 | 内容 |
|------|------|
| 認証方式 | Supabase Auth（Email + Password）。アプリ独自の認証処理は行わない |
| セッション維持 | Supabase Auth（JWT: Access Token 1時間 / Refresh Token 自動ローテーション）。`@supabase/ssr` により Cookie で自動管理 |
| パスワードポリシー | 英数字混在 8文字以上（Supabase Auth 側で設定） |
| パスワードハッシュ | Supabase Auth 内部で bcrypt 処理（アプリ側実装不要） |
| アカウントロック | Supabase Auth の Rate Limiting による制御（アプリ独自のロック管理は行わない） |
| 初回ログイン | 管理者が発行した仮パスワードでログイン後、ダッシュボードから変更 |
| パスワード再発行 | スタッフがパスワードを忘れた場合、管理者が仮パスワードを再発行する |
| ログアウト | Supabase Auth のセッションを破棄し、Cookie をクリアして、ログイン画面にリダイレクトする |

> **認証方針:** 認証処理は全面的に Supabase Auth に委任する。パスワードハッシュ、ログイン失敗回数、アカウントロック等をアプリケーション独自テーブルで管理しない（二重管理防止）。
> ページアクセスの未認証時はログイン画面へリダイレクトし、APIアクセスの未認証時は `401 UNAUTHORIZED` を返す。

## 2. スタッフマスタ管理

### 2.1 一覧表示

- 有効スタッフの一覧をチーム別（A/B）で表示
- 検索：氏名、スタッフコードでの絞り込み
- ソート：氏名（50音順）、経験年数

### 2.2 登録・編集

| フィールド | 型 | 必須 | バリデーション |
|------------|------|:--:|----------------|
| スタッフコード | 文字列 | ○ | 英数字 2〜10文字 / 一意制約。業務上の識別子（例: `N001`） |
| 氏名 | 文字列 | ○ | 1〜30文字 |
| 経験年数 | 整数 | ○ | 1〜50 |
| 所属チーム | 列挙型 | ○ | A / B |
| 夜勤可否 | 真偽値 | ○ | デフォルト: 可 |
| メールアドレス | 文字列 | ○ | メール形式 / 一意制約 |

### 2.3 削除（論理削除）

- 有効フラグを `false` に変更。データベース上のレコードは保持
- 論理削除済みスタッフは未確定のタームのシフトから自動除外
- 管理者は論理削除済み一覧を参照可能（復元も可能）

## 3. ターム設定

### 3.1 ターム登録

- 開始日を指定すると、終了日（開始日 + 27日）を自動計算
- **年度（`fiscal_year`）** は開始日から自動導出する（4月〜翌3月。例: 2026年4月開始 → 年度2026、2027年2月開始 → 年度2026）
- 年度ごとに一括登録する機能を提供（13ターム分）
- 一括登録時、開始日は任意の日付を指定可能（月曜始まりの強制はしない）
- 13ターム × 28日 = 364日。残り1〜2日は次年度の第1タームで吸収する（年度間にギャップが生じてもよい）
- 個別追加登録も可能。ターム間にギャップ（空白期間）があることを許容する
- ターム期間の重複チェックを実施（ギャップは許容するが重複は禁止）
- `draft` 状態のタームのみ削除可能（それ以降のステータスでは削除不可。UIでは「削除」と表示）

### 3.2 ターム状態管理

各タームは以下のステータスを持つ。

| ステータス | 説明 |
|-----------|------|
| `draft` | 初期状態。希望休受付前 |
| `collecting` | 希望休受付中 |
| `generating` | シフト生成中 |
| `adjusting` | シフト生成完了・手動調整中 |
| `confirmed` | シフト確定済み。スタッフに公開 |

状態遷移:
```
draft → collecting → generating → adjusting → confirmed
                         ↑            ↑           |
                         |            +-----------+
                    （生成失敗時）     （差し戻し可能）
```

> **遷移ルール補足:**
> - `collecting` → `generating`: 管理者がシフト表画面で「自動生成」ボタンを押した時点で遷移する（自動遷移はしない）。
> - `generating` → `adjusting`: 生成が正常完了した場合に自動遷移する。
> - `generating` → `collecting`: 生成が例外エラーで失敗した場合にロールバックする。タイムアウトの場合は途中結果を返却し `adjusting` に遷移する。
> - `confirmed` 状態のシフトを再編集するには、必ず `adjusting` への差し戻しが必要。`confirmed` 状態では直接編集できない。
>
> **`generating` スタック復旧:**
> サーバークラッシュ等で `generating` 状態のままステータスが固まった場合に備え、以下の復旧手段を提供する。
> - `generating` に遷移してから **5分以上** 経過したタームに対し、管理者は `PUT /api/terms/:id/status` で `collecting` への強制ロールバックを実行できる（5分未満の場合は拒否）。
> - ヘルスチェック Cron（5分間隔）で `generating` 状態が10分以上経過しているタームを検出した場合、自動的に `collecting` にロールバックし、管理者に通知ログを記録する。

### 3.3 祝日マスタ

- 日本の祝日データをシステムに内蔵（「国民の祝日に関する法律」に基づく）
- 祝日データは**暦年**単位で管理し、ターム画面・シフト生成では対象期間の日付範囲で参照する
- 病院独自の休日（記念日等）を管理者が追加可能
- **初期投入:** シードスクリプト（`prisma/seed.ts`）で2026〜2027年の祝日を投入する。春分の日・秋分の日は暦要項の確定値を手動で定義する
- **年次更新:** MVP では管理者が祝日管理画面（`POST /api/holidays`）から翌年分を手動登録する運用とする。将来的には外部API連携や管理画面での一括登録を検討する

### 3.4 祝日・代休ルール

- 祝日に勤務したスタッフには代休（`holiday_off`）を1日付与する
- 代休は同一ターム内で消化することを原則とする
- 代休は週休（H5 の8回）には含めず、別枠として管理する
- シフト自動生成時、祝日の勤務者に対して代休を自動配置する

## 4. 希望休入力

### 4.1 スタッフ側操作

- 対象タームのカレンダーUIから希望日を選択（最大3日）
- 申請済みの希望休の取消・変更が可能（締切前のみ）
- 申請状況の確認画面を提供

### 4.2 管理者側操作

- 各タームの希望休締切日を設定
- `request_deadline` は `DATE` とし、**サーバー時刻のJSTにおける当日 23:59:59 まで有効**とする
- 締切判定はAPI側で `現在のJST日付 > request_deadline` をリクエスト時にチェックする方式とする（バッチ処理やステータス自動遷移は行わない）
- 全スタッフの希望休申請状況を一覧確認
- 管理者は締切後もスタッフの希望休を代理入力・修正可能

## 5. シフト自動生成

### 5.1 生成トリガー

- 管理者がシフト表画面で「自動生成」ボタンを押下
- 生成前に入力データ（スタッフ数・希望休）のバリデーションを実施
- 有効スタッフ数が `ceil((min_day_staff + 3 + 3) × 28 / (20 - estimated_holidays))` を下回る場合はエラー。`estimated_holidays` はターム期間内の祝日数とする
- デフォルト設定（`min_day_staff = 7`、祝日0日）では、必要総勤務枠 `13枠 × 28日 = 364枠` に対して1人あたり最大20勤務日のため、**有効スタッフ19名以上**を生成前提とする。祝日が多いタームでは代休による追加休暇で1人あたりの勤務可能日数が減るため、必要スタッフ数が増加する
- 夜勤可能スタッフ数が `ceil((3 + 3) × 28 / (20 - estimated_holidays))` を下回る場合はエラー
- デフォルト設定（祝日0日）では夜勤枠 `6枠 × 28日 = 168枠` に対して、**夜勤可能スタッフ9名以上**を生成前提とする
- 組み合わせ制約 H18 を成立させるため、チームA・Bそれぞれに**夜勤可能スタッフ3名以上**を必要とする

### 5.2 ハード制約（必須遵守）

> **MVP段階分け:** ハード制約はMVPフェーズ1（必須）とMVPフェーズ2（準ハード）に分類する。フェーズ1は生成時に必ず遵守する。フェーズ2はMVP初期リリースでは「警告扱い」とし、生成は許可するが違反は赤色警告で表示する。フェーズ2の完全遵守はMVP後の改善で対応する。

#### 人員配置制約【フェーズ1: MVP必須】

| ID | 制約内容 |

| H1 | 準夜勤は毎日3名を配置 |
| H2 | 深夜勤は毎日3名を配置 |
| H3 | 日勤は毎日最低7名を配置（管理者設定可能） |
| H4 | 夜勤不可スタッフは準夜勤・深夜勤に配置しない |

#### 週休・休日制約

### 5.3 ソフト制約（推奨）

| ID | 制約内容 | 優先度 |
|----|---------|:---:|
| S1 | 準夜・深夜の回数をスタッフ間で均等配分（差±1回以内が目標） | 高 |
| S2 | 各スタッフの準夜回数と深夜回数のバランスを取る | 中 |
| S3 | 夜勤の基本パターンは「深夜→準夜」のセットを優先 | 中 |
| S4 | 希望休を可能な限り反映 | 高 |

### 5.4 生成結果

- 生成結果はタームのステータスを `adjusting` に変更し、シフト表画面に表示
- 生成結果は current working copy として `shift_entries` に即時保存する（スナップショットは作成しない）
- 満たせなかったソフト制約の一覧を表示
- 再生成ボタンで別パターンの再生成が可能
- **再生成時の保護:** `adjusting` 状態で再生成を実行する場合、再生成前に現在の `shift_entries` をスナップショットとして自動保存する。これにより手動調整済みデータの消失を防止する

## 6. 手動調整・警告機能

### 6.1 手動編集

- シフト表のセルをクリックし、勤務種別（日・準・深・休・代）を選択して変更
- **編集はフロントエンドの Zustand ストアにのみ反映し、「保存」ボタン押下まで一切のサーバーAPIを呼び出さない**（クライアント側ステート管理）
- 制約チェックはクライアント側でリアルタイム実行する（変更ごとにサーバーへ送信しない）
- Undo / Redo 機能を提供（直近20操作まで、未保存のフロントエンド操作履歴のみ）
- `requested_off`（希）は希望休申請由来の表示値であり、手動編集で直接選択はできない
- 保存API（`POST /api/terms/:termId/shifts/save`）は、対象タームの current working copy を**完全置換**する
- 保存時は有効スタッフ分の全シフトエントリを送信し、**未送信行は削除**される。無効化済みスタッフの行は current working copy の保存対象から**自動除外**される

> **設計方針:** セル変更ごとにサーバーへ送信する設計（逐次保存）は採用しない。全編集操作はフロントエンドで保持し、「保存」時に `POST /api/terms/:termId/shifts/save` で一括永続化する。保存は current working copy の完全置換として扱い、無効化済みスタッフはサーバー側で自動除外する。これにより Undo/Redo・isDirty・制約違反表示の責務がフロントエンドに集中し、実装が明確になる。

### 6.1a 楽観的ロック

- シフト表の保存時に楽観的ロック（Optimistic Locking）を採用する
- ターム単位の専用カラム `lock_version`（整数、初期値 1）を使用する
- シフトデータ取得時に `lock_version` を返却する
- 保存リクエストに `lock_version` を含め、サーバー側で現在の `lock_version` と一致するかチェックする
- 一致した場合は保存後に `lock_version` をインクリメントする（+1）
- 不一致の場合は `409 CONFLICT`（OPTIMISTIC_LOCK_ERROR）を返し、クライアントに最新データの再取得を促す
- MVP では管理者1名での運用を主想定とするが、複数端末・複数タブでの同時操作に備えて実装する

> **Undo/Redo と履歴の役割分離:**
> - **Undo/Redo**: フロントエンドの未保存編集状態のみを対象。Zustand ストア内で管理。保存するとスタックはリセットされる。
> - **スナップショット（§8）**: 「保存」操作時にサーバーに永続化される世代履歴。任意の世代にロールバック可能。
> - **操作ログ**: 変更者・変更日時・変更箇所の記録（非機能要件の操作ログで管理）。

### 6.2 制約違反警告

| レベル | 表示色 | 対象 | 保存 |
|--------|--------|------|------|
| エラー | 赤 | フェーズ1必須制約違反 | 保存可能だが `confirmed` への確定は不可 |
| 警告 | 黄 | フェーズ2制約違反 / ソフト制約違反 | 保存可能 |

- 違反一覧をサマリーパネルで表示（違反ID・対象スタッフ・違反内容）
- 違反セルをクリックすると該当セルにスクロール

### 6.3 勤務時間の計算ルール

| 勤務種別 | 勤務時間帯 | 実働時間（休憩除く） |
|---------|----------|:------------------:|
| 日勤 | 8:30〜17:15 | 7.75時間（休憩1時間） |
| 準夜勤 | 16:30〜翌1:15 | 7.75時間（休憩1時間） |
| 深夜勤 | 0:30〜9:15 | 7.75時間（休憩1時間） |

- 1タームの基準勤務時間: **155時間**（実働7.75時間 × 20勤務日）
- 個別標準勤務時間: `155時間 - (holiday_off 日数 × 7.75時間)`
- 統計表示では各スタッフの合計勤務時間と**個別標準勤務時間**との差分を表示する

### 6.4 統計情報リアルタイム表示

- 各スタッフの勤務時間合計（個別標準勤務時間との差分）
- 各スタッフの準夜勤回数・深夜勤回数
- 週休回数・法定週休回数
- 各日の勤務帯別配置人数
- チーム別の配置バランス

## 7. シフト表出力

### 7.1 画面表示

- 1ターム28日間のグリッド（行: スタッフ、列: 日付）
- 各セルに勤務記号（日・準・深・休・祝・代・希）を表示
- 土曜: 薄青背景 / 日曜: 薄赤背景 / 祝日: 薄桃背景
- ハード制約違反セル: 赤枠 / ソフト制約違反セル: 黄枠
- 右端列にスタッフ別統計情報（勤務時間・夜勤回数・週休数）

### 7.2 PDF出力

- A3横向き、1ターム1ページ
- グリッド形式のレイアウトは画面表示と同等
- ヘッダー: 病棟名・ターム期間・出力日時
- フッター: ページ番号
- **出力対象:** `adjusting` または `confirmed` ステータスのタームのみ出力可能（`draft` / `collecting` / `generating` では出力ボタン非活性）

## 8. シフト表編集履歴（スナップショット）

- 「保存」ボタン押下時にサーバーにスナップショットを永続化する（直近5世代分保持）
- version は連番でインクリメントする（1, 2, 3, ...）。6世代目の作成時に最古の世代を自動削除し、常に最新5世代を保持する
- 各世代のスナップショットを保存し、任意の世代にロールバック可能
- 変更者・変更日時を記録（変更箇所の詳細は操作ログとして別途記録）

> **注意:** §6.1 の Undo/Redo はフロントエンドの未保存操作を対象とする機能であり、このスナップショットとは独立して動作する。

| 5 | ターム設定 | `/admin/terms` | 管理者 | ターム登録・管理 |
| 6 | 希望休入力 | `/requests` | スタッフ | 希望休申請 |
| 7 | シフト表（編集） | `/admin/shifts/:termId` | 管理者 | シフト生成・編集・警告 |
| 8 | シフト表（閲覧） | `/shifts/:termId` | スタッフ | 確定シフト閲覧 |
| 9 | パスワード変更 | `/settings/password` | 全ユーザー | パスワード変更 |

> **レスポンシブ対応:** スタッフ向け画面（No.3, 6, 8）はスマートフォンからのアクセスを考慮し、レスポンシブデザインで実装する。管理者向け画面はPC（1280×720以上）のみを対象とする。

## 10. 設計上の明示事項

### 10.1 管理者と Staff の関係

MVP では **管理者（師長）も必ず `staffs` テーブルのレコードを持つ**（`users` : `staffs` = 1:1 を全ユーザーに対して維持する）。

- 師長がシフト対象の看護師でもある運用を前提とする
- 純管理者専用アカウント（`staffs` レコードなし）は MVP スコープ外とする
- これにより `staffs.user_id` は全ユーザーで NOT NULL を維持できる

> この方針は希望休API・スタッフ一覧APIの実装に影響する（管理者も staff として取得対象に含まれる）。将来的に純管理者アカウントが必要になった場合は `staffs.user_id` を nullable に変更する。

### 10.2 管理者のシフト・希望休操作

- **シフト自動生成:** 管理者の staffs レコードもシフト対象に含む（有効スタッフとして扱う）
- **希望休入力:** 管理者は自身の希望休をスタッフ向け画面（`/requests`）ではなく、管理者画面の希望休一覧（`GET /api/terms/:termId/requests`）から代理入力の形式で登録する。管理者は締切後も自身の希望休を変更可能
- **画面アクセス:** 管理者は `/admin/*` 配下の画面にアクセスする。スタッフ向け画面（`/home`, `/requests`, `/shifts/:termId`）には管理者はアクセスしない設計とする（Middleware でリダイレクト）

## 11. MVP スコープ外（将来検討）

以下の機能は MVP では対応しない。将来のフェーズで検討する。

| 機能 | 備考 |
|------|------|
| メール通知 | 希望休受付開始・シフト確定・パスワード再発行等の通知。MVP ではアプリ内の画面で確認する運用 |
| 複数病棟・複数部署対応 | 現状は単一病棟前提。テーブル設計にテナント概念を入れない |
| データエクスポート/インポート | スタッフマスタのCSVインポート等 |
| 代休の次ターム繰越 | 同一ターム内消化を原則とし、繰越は対応しない |
