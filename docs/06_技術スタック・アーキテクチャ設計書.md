# 技術スタック・アーキテクチャ設計書

## NHO 看護師シフト管理アプリ — MVP

---

## 1. この文書の位置づけ

- 本書は MVP の**技術選定方針**と**責務分割**を定義する
- 実装詳細、API 入出力、DB スキーマ、業務制約の正本はそれぞれ `04_データ設計書.md`、`05_API設計書.md`、`07_業務ルール詳細定義書.md` を参照する
- ディレクトリ構成、環境変数、IaC、ライブラリ細部は実装時に変更し得るため、本書では固定しない

---

## 2. 技術スタック方針

| レイヤー | 採用方針 | 備考 |
|---------|---------|------|
| フロントエンド | React 系フレームワークを採用 | 管理画面とスタッフ画面を同一アプリで提供できること |
| UI | コンポーネント再利用可能な UI 基盤を採用 | 表形式 UI と状態表示の実装効率を重視 |
| バックエンド | 同一リポジトリ内で API を提供できる構成を採用 | 画面と API の整合を保ちやすいこと |
| DB | マネージド PostgreSQL を採用 | 日付・集計・制約検証に強いこと |
| 認証 | Supabase Auth に委任 | パスワードやセッションをアプリで二重管理しない |
| ORM / クエリ層 | 型安全な DB アクセス層を採用 | スキーマ変更時の追従性を重視 |
| バリデーション | フロント・バックで共有可能なスキーマ定義を採用 | API と画面の解釈差を減らす |
| PDF 出力 | サーバーサイドで生成。軽量ライブラリ（`@react-pdf/renderer` または `pdfkit`）を使用 | Vercel Lambda のサイズ制限を考慮し、Puppeteer/Playwright は使用しない。A3横向きグリッドレイアウトの実装容易性で選定する |

> **補足:** 採用候補としては Next.js + Route Handlers、Prisma、Zod、Supabase を前提とするが、MVP 計画書としてはバージョンや補助ライブラリまで固定しない。

## 3. アーキテクチャ全体像

```
利用者ブラウザ
  ↓
Web アプリ
  ├─ 管理者 UI
  ├─ スタッフ UI
  └─ API 層
       ├─ 認証・認可
       ├─ ターム / 希望休 / シフト / 祝日 API
       ├─ シフト生成オーケストレーション
       └─ PDF 出力
  ↓
データ層
  ├─ アプリケーション DB
  └─ 認証基盤
```

### 3.1 責務分割

| 層 | 主責務 |
|----|------|
| フロントエンド | 画面表示、入力補助、未保存編集状態、クライアント側の軽量チェック |
| API 層 | 認可、永続化、整合チェック、生成処理の起動、統計計算結果の返却 |
| 生成エンジン | 制約を考慮したシフト案の構築、修復、評価。制約チェックロジックはフロントエンドと共有する（§3.3 参照） |
| DB | ターム、スタッフ、希望休、シフト、祝日、履歴、監査ログの保存 |
| 認証基盤 | ログイン、セッション、パスワード管理 |

### 3.3 制約チェックロジックの共有方針

シフトのハード制約（H1〜H19）・ソフト制約（S1〜S4）の判定ロジックは、フロントエンド（リアルタイム制約チェック）とバックエンド（保存時の制約チェック・生成エンジン）の両方で必要となる。二重実装による不整合を防ぐため、以下の方針を採用する。

- **制約チェック関数を `src/lib/constraints/` に純粋関数として実装する**。DB 依存・フレームワーク依存を排除し、シフトデータ配列と設定値のみを入力とする
- フロントエンド（Zustand ストア内のリアルタイムチェック）とバックエンド（API 保存時・生成エンジン内）の両方から同一の関数をインポートして使用する
- 制約関数は副作用を持たず、違反情報の配列を返す純粋関数とする。テストも共通のテストスイートで行う
- 統計計算（勤務時間・夜勤回数等）も同様に `src/lib/statistics/` に配置し、フロント・バックで共有する

```
src/lib/
  constraints/
    hard-constraints.ts    // H1〜H19 の判定関数
    soft-constraints.ts    // S1〜S4 の判定関数
    types.ts               // 制約違反の型定義
    index.ts               // エクスポート
  statistics/
    shift-statistics.ts    // 勤務時間・夜勤回数等の計算
```

## 4. バックエンド設計方針

### 4.1 API 設計の考え方

- 画面ごとの都度保存ではなく、シフト編集は**未保存状態をクライアントに保持し、保存時に一括反映**する
- 生成・保存・確定など副作用の大きい処理は明示 API とする
- 権限判定は API 層で必ず実施する
- 締切判定、確定可否判定、生成前バリデーションは API 層を正とする

### 4.2 認証・認可

- 認証は Supabase Auth に委任する
- アプリ側はユーザーの業務属性とロールに基づく認可のみを担う
- 未認証、権限不足、締切超過、状態不整合は API レベルで明示的にエラー化する

### 4.3 永続化

- ターム、希望休、シフト、祝日、履歴、監査ログは PostgreSQL に保存する
- シフト保存はターム単位で整合を取る
- 同時編集対策としてターム単位の楽観的ロックを採用する

---

## 5. フロントエンド設計方針

### 5.1 状態管理

- シフト表の未保存編集状態はクライアント側ストアで保持する
- Undo / Redo は未保存状態に限定する
- 保存成功時に未保存履歴はリセットする

### 5.2 検証と表示

- セル編集時は即時に軽量チェックを行い、違反候補を可視化する
- 最終的な制約判定、統計、保存可否はサーバー返却値を正とする
- 画面は「編集中の利便性」と「サーバーが保証する整合性」を分離して扱う

### 5.3 帳票

- PDF はサーバー側で生成する
- 帳票レイアウトは画面と概ね整合させるが、出力都合による差異は許容する

## 6. シフト生成エンジン方針

### 6.1 目的

- ハード制約を優先して満たすシフト案を生成する
- ソフト制約はスコアリング対象として改善する
- 完全最適解の保証よりも、MVP として実用的な時間内に解を返すことを優先する

### 6.2 入力前提

- 必要人数の妥当性は生成前バリデーションで確認する
- デフォルト条件では有効スタッフ19名以上、夜勤可能スタッフ9名以上、かつチームA/Bそれぞれ3名以上を前提とする
- この前提を満たさない場合は生成を開始しない

### 6.3 実装自由度

- 探索手法はグリーディ、局所探索、ヒューリスティクスの組み合わせを基本とする
- ただし各フェーズの秒数配分、内部データ構造、探索順序は実装で調整可能とする
- タイムアウト時は最善の中間結果を返す方針とし、内部アルゴリズムの固定までは行わない

## 7. インフラ・実行環境方針

### 7.1 実行環境の要件

- Web アプリと API を同一デプロイ単位で運用できること
- シフト自動生成に対して**約60秒級のサーバー実行時間**を確保できること
- PDF 生成と DB 接続を安定運用できること

> **Vercel 実行時間制限への対応:**
> - Vercel Pro プランのサーバーレス関数は最大60秒（デフォルト10秒）。シフト生成 API（55秒タイムアウト）を動作させるため、Route Handler に `export const maxDuration = 60` を設定する
> - コールドスタートのオーバーヘッド（数秒）を考慮し、生成エンジンの内部タイムアウトは **55秒** とする（60秒 - コールドスタートマージン）
> - 生成中は長時間コネクションを占有するため、Prisma のコネクションプール設定（`connection_limit`）に注意する。生成処理ではトランザクションを必要最小限に留め、結果保存時のみトランザクションを使用する

### 7.2 デプロイ方針

- プレビュー環境と本番環境を分ける
- マージ前に自動テストを実行する
- 具体的なホスティング設定値や CI 設定ファイルは運用資料または実装リポジトリ側で管理する

### 7.3 機密情報管理

- 秘密情報はホスティング基盤のシークレット管理機能で扱う
- 本書に実値や接続文字列例は保持しない

## 8. セキュリティ方針

| 項目 | 方針 |
|------|------|
| 認証 | Supabase Auth を利用し、アプリ独自のパスワード管理は行わない |
| 認可 | ロールと対象リソースに応じて API 層で判定する |
| 入力検証 | API 境界で必須。クライアント側検証は補助扱い |
| 監査 | 重要操作は操作ログに記録する |
| 通信 | HTTPS 前提 |
| 競合制御 | 楽観的ロックで多重保存を防ぐ |

---

## 9. MVP で固定しない事項

以下は実装判断の余地を残し、本書では固定しない。

- 具体的なディレクトリ構成
- 状態管理ライブラリのインターフェース詳細
- 環境変数名の最終形
- デプロイ設定ファイルの中身
- 生成エンジン内部の秒数配分
- セキュリティヘッダーやミドルウェアの実装細部

---

## 10. 今後の拡張余地

- 生成エンジンの高度化
- 帳票テンプレートの複数化
- 通知機能の追加
- 認可モデルの細分化
- RLS の本格活用

MVP ではこれらを前提化せず、現在必要な責務分割と整合性確保を優先する。
