# ワークフロー設計書

## NHO 看護師シフト管理アプリ — MVP

---

## 1. 全体業務フロー

```
┌──────────────────────────────────────────────────────────────────┐
│  1. ターム設定      管理者がターム開始日を登録                       │
│       ↓                                                          │
│  2. 希望休受付開始   管理者が締切日を設定し、受付ステータスに変更      │
│       ↓                                                          │
│  3. 希望休入力      スタッフが希望休を申請（最大3日/ターム）          │
│       ↓                                                          │
│  4. 希望休締切      締切条件到来でスタッフ入力をAPIでブロック           │
│       ↓                                                          │
│  5. シフト自動生成   管理者が生成を実行                              │
│       ↓                                                          │
│  6. 手動調整       管理者がシフト表を確認・修正                      │
│       ↓                                                          │
│  7. シフト確定      管理者が確定。スタッフに公開                     │
│       ↓                                                          │
│  8. シフト閲覧      スタッフが確定シフトを閲覧                      │
│       ↓                                                          │
│  9. PDF出力        管理者がPDF出力し印刷・配布                      │
└──────────────────────────────────────────────────────────────────┘
```

## 2. ターム設定フロー

```
管理者
  │
  ├─ [年度一括登録]
  │    │
  │    ├─ 年度開始日（4月第1月曜日等）を入力
  │    ├─ システムが28日単位で13ターム分を自動計算
  │    ├─ 重複チェック → 既存タームとの重複があればエラー
  │    └─ 確認画面で一覧表示 → 登録確定
  │
  └─ [個別登録]
       │
       ├─ 開始日を入力
       ├─ 終了日が自動計算（開始日 + 27日）
       ├─ 重複チェック
       └─ 登録確定
```

## 4. シフト自動生成フロー

```
管理者
  │
  ├─ シフト表画面で「自動生成」ボタン押下
  │
  ├─ [入力バリデーション]
  │    ├─ 有効スタッフ数 ≧ ceil((min_day_staff + 6) × 28 / (20 - 祝日数))？
  │    │    └─ 既定値（祝日0日）では 19名以上（364勤務枠 ÷ 1人最大20勤務日）
  │    ├─ 夜勤可能スタッフ数 ≧ ceil(6 × 28 / (20 - 祝日数))？
  │    │    └─ 既定値（祝日0日）では 9名以上（168夜勤枠 ÷ 1人最大20勤務日）
  │    ├─ チームA/Bそれぞれ夜勤可能者が3名以上？
  │    └─ NG → エラー表示、生成中止
  │
  ├─ [生成エンジン実行]（詳細は後述）
  │    ├─ ステータスを「generating」に変更
  │    ├─ フェーズ1: 夜勤配置（深夜→準夜のセットを優先）
  │    ├─ フェーズ2: 週休配置（8回/ターム、制約を満たすよう配置）
  │    ├─ フェーズ3: 祝日勤務者の確認 → 代休配置（週休とは別枠）
  │    ├─ フェーズ4: 日勤配置（残りの勤務日を日勤で埋める）
  │    ├─ フェーズ5: 制約検証・修復
  │    ├─ フェーズ6: ソフト制約の最適化
  │    ├─ 55秒到達時は最善結果を返却し「adjusting」に遷移
  │    └─ 例外エラー発生時はステータスを「collecting」にロールバック
  │
  ├─ [生成結果表示]
  │    ├─ 生成結果を current working copy として永続化
  │    ├─ ステータスを「adjusting」に変更
  │    ├─ シフト表をグリッド表示
  │    ├─ 警告（フェーズ2制約・ソフト制約）の一覧を表示
  │    ├─ 統計情報を表示
  │    └─ タイムアウト時は「途中結果」である旨を表示
  │
  └─ [再生成]
       ├─ 「再生成」ボタンで別パターンを生成
       ├─ 再生成前に現在のシフトデータをスナップショットとして自動保存
       └─ 乱数シードを変更して再実行
```

## 5. シフト生成エンジン処理フロー（詳細）

```
入力: スタッフ一覧、ターム情報、希望休データ、祝日データ、前タームのシフトデータ（確定済みの場合）

Step 1: 初期化
  ├─ 生成前バリデーションで必要人数を確認
  │    ├─ 有効スタッフ数: ceil((min_day_staff + 6) × 28 / (20 - 祝日数)) 以上
  │    ├─ 夜勤可能スタッフ数: ceil(6 × 28 / (20 - 祝日数)) 以上
  │    └─ チームA/Bの夜勤可能スタッフ: 各3名以上
  ├─ 28日 × Nスタッフのシフトテーブルを空で初期化
  ├─ 希望休をテーブルに反映（「休」で確定）
  ├─ 夜勤不可スタッフを夜勤候補から除外
  └─ 前タームが確定済みの場合、最終3日間のシフトデータを読み込み
     （ターム境界での夜勤制約 H11〜H16 の連続性チェック用）

Step 2: 夜勤配置
  ├─ 夜勤可能スタッフの一覧を取得
  ├─ 目標夜勤回数を計算（均等配分）
  ├─ 各日に対して:
  │    ├─ 深夜勤3名を選出（H17: 経験者混在、H18: チーム混在）
  │    ├─ 準夜勤3名を選出（同上制約）
  │    ├─ H11〜H16の夜勤制約をチェック
  │    └─ 制約違反があれば別の候補でリトライ
  └─ S3: 深夜→準夜のセットパターンを優先配置

Step 3: 週休配置
  ├─ 各スタッフに8回の週休を配置
  ├─ H6: 週休間隔を最大5日に制御
  ├─ H7: 土日連休を1回以上確保
  ├─ H8: 平日連休を1回以上確保
  ├─ H9: 48時間以上の休み（準夜明けでない連休）を確保
  ├─ H10: 法定週休を4回以上確保
  └─ 夜勤配置済み日を避けて配置

Step 4: 代休配置
  ├─ 祝日マスタと生成済みシフト全体を突合
  ├─ 祝日に勤務するスタッフを特定（日勤・準夜勤・深夜勤を含む）
  ├─ 各対象スタッフに holiday_off を1日配置（同一ターム内）
  ├─ 配置先は夜勤・希望休と重複しない日を選択
  └─ 配置困難な場合はソフト制約違反として記録（生成は継続）

Step 5: 日勤配置
  ├─ 残りの未割り当て日を日勤で埋める
  ├─ H3: 各日の日勤者数 ≧ 7名を確認
  └─ 不足があれば週休位置を調整

Step 6: 制約検証・修復
  ├─ 全ハード制約を再検証
  ├─ 違反があれば局所的な入れ替えで修復を試行
  └─ 修復不可の場合は Step 2 からリトライ（最大5回）

Step 7: ソフト制約の最適化
  ├─ S1: 夜勤回数の均等性をスコアリング
  ├─ S2: 準夜/深夜バランスをスコアリング
  ├─ S4: 希望休反映率をスコアリング
  └─ スコアの改善余地があればスワップを試行

エラーハンドリング:
  ├─ タイムアウト（55秒）到達 → 最善結果を返却し adjusting に遷移
  └─ 例外エラー発生 → ステータスを collecting にロールバックし、エラーを返却
```

## 6. 手動調整フロー

> **設計方針:** 保存方式の正本は `01_機能要件定義書.md` と `05_API設計書.md` を参照。本フローでは、セル変更ごとにサーバーAPIを呼び出さず、「保存」ボタン押下時のみ `POST /api/terms/:termId/shifts/save` を呼び出す前提で扱う。

```
管理者
  │
  ├─ シフト表のセルをクリック
  │
  ├─ 勤務種別選択ダイアログ表示
  │    └─ 日 / 準 / 深 / 休 から選択
  │
  ├─ [フロントエンド側ステート更新]（サーバー通信なし）
  │    ├─ Zustand ストアのシフトデータを更新
  │    ├─ isDirty フラグを true に設定
  │    └─ Undo/Redo スタックに操作を追加
  │
  ├─ [クライアント側即時制約チェック]（サーバー通信なし）
  │    ├─ 変更対象スタッフの全制約を再評価
  │    ├─ 変更対象日の配置人数を再計算
  │    │
  │    ├─ ハード制約違反あり？
  │    │    └─ YES → 赤色警告を表示（違反内容を明示）
  │    │          変更は適用されるが警告は消えない
  │    │
  │    └─ ソフト制約違反あり？
  │         └─ YES → 黄色警告を表示
  │
  ├─ 統計情報をリアルタイム更新（クライアント計算）
  │    ├─ 勤務時間合計
  │    ├─ 夜勤回数
  │    ├─ 週休回数
  │    └─ 各日配置人数
  │
  ├─ 「保存」ボタンで変更を永続化（POST /api/terms/:termId/shifts/save 呼び出し）
  │    ├─ フロントエンドの有効スタッフ分の全シフトエントリを一括送信
  │    ├─ 保存は current working copy の完全置換として扱われ、未送信行は削除される
  │    ├─ 無効化済みスタッフの行はサーバー側で自動除外される
  │    ├─ 楽観的ロック: リクエストに lock_version を含める
  │    ├─ 成功: isDirty をリセット、Undo/Redo スタックをクリア、新 lock_version を保持
  │    ├─ 409 OPTIMISTIC_LOCK_ERROR: 最新データ再取得を促すダイアログを表示
  │    └─ 編集履歴として現在の状態をスナップショット保存（サーバー側で自動実行）
  │
  └─ 必要に応じて Undo / Redo（未保存の操作のみ対象）
```

## 7. シフト確定フロー

```
管理者
  │
  ├─ シフト表画面で「確定」ボタン押下
  │
  ├─ [確定前チェック]
  │    ├─ フェーズ1必須制約違反が残っている場合
  │    │    └─ エラー表示「必須制約違反が解消されるまで確定できません」
  │    └─ 警告のみの場合（フェーズ2制約・ソフト制約）
  │         └─ 確認ダイアログ「シフト表を確定しますか？
  │            確定後スタッフに公開されます。」
  │
  ├─ タームステータスを「confirmed」に変更
  │
  ├─ スタッフ側のシフト閲覧画面に公開
  │
  └─ [差し戻し]
       ├─ 確定後に修正が必要な場合
       ├─ 「差し戻し」ボタンでステータスを「adjusting」に戻す
       └─ スタッフ閲覧画面から非公開に
```

## 8. PDF出力フロー

```
管理者
  │
  ├─ シフト表画面で「PDF出力」ボタン押下
  │
  ├─ サーバーサイドでPDF生成
  │    ├─ A3横向き、1ターム1ページ
  │    ├─ ヘッダー: 病棟名・ターム期間・出力日時
  │    ├─ 本体: シフトグリッド + 統計情報
  │    └─ フッター: ページ番号
  │
  └─ ブラウザにPDFをダウンロード
```

## 9. 状態遷移図（ターム）

```
        ┌─────────┐
        │  draft  │  初期状態
        └────┬────┘
             │ 希望休受付開始（管理者手動）
             ▼
      ┌──────────────┐
      │  collecting  │  希望休受付中
      └──────┬───────┘
             │ 自動生成ボタン押下（管理者手動）
             ▼
      ┌──────────────┐
      │  generating  │  生成処理中
      └──┬───────┬───┘
         │       │ 生成失敗（例外エラー）or スタック復旧（5分超経過時）
         │       ▼
         │  ┌──────────────┐
         │  │  collecting  │  ← ロールバック
         │  └──────────────┘
         │ 生成完了 or タイムアウト（途中結果）
         ▼
      ┌──────────────┐
      │  adjusting   │  手動調整中
      └──────┬───────┘
             │ 確定（管理者手動）
             ▼
      ┌──────────────┐
      │  confirmed   │  確定・公開
      └──────┬───────┘
             │ 差し戻し（管理者手動）
             ▼
      ┌──────────────┐
      │  adjusting   │  再調整
      └──────────────┘
```
