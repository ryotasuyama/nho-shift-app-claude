## 3. ターム管理

### 3.1 ターム登録

| フィールド / 条件 | ルール | エラーコード | メッセージ |
|-----------|--------|-----------|---------|
| start_date | 必須 | REQUIRED | 開始日を入力してください |
| start_date | 有効な日付形式 | INVALID_FORMAT | 日付の形式が正しくありません |
| start_date | 過去日でないこと | PAST_DATE | 過去の日付は指定できません |
| 期間重複 | 既存タームと重複しない | CONFLICT | 指定された期間は既存のタームと重複しています |

### 3.2 ターム編集

| フィールド / 条件 | ルール | エラーコード | メッセージ |
|-----------|--------|-----------|---------|
| request_deadline | 有効な日付形式 | INVALID_FORMAT | 日付の形式が正しくありません |
| request_deadline | 有効な日付であれば設定可能。`start_date` より前を推奨するが、ターム期間内・期間後もサーバーは受理する | — | — |
| min_day_staff | 1〜20 の整数 | OUT_OF_RANGE | 日勤最低人数は1〜20の範囲で設定してください |

> **補足:** `request_deadline` はシフト対象期間（`start_date`〜`end_date`）より前に設定するのが通常の業務運用。サーバーは有効な日付であれば受理するが、UI は以下の警告方針を採用する。
> - `request_deadline < start_date`: 警告なし
> - `start_date <= request_deadline <= end_date`: 警告表示「締切日がターム開始日以降です。希望休の募集期間が短くなる可能性があります」
> - `request_deadline > end_date`: 警告表示「締切日がターム終了日を超えています。運用上の設定を確認してください」
>
> いずれの警告も **入力は許可** し、サーバーエラーにはしない。未設定のままでは `collecting` への遷移が不可（§3.4 参照）。

> **締切判定ルール:** `request_deadline` は DATE として保持し、スタッフ操作は `現在のJST日付 > request_deadline` になった時点で締切後と判定する。締切日当日の23:59:59 JSTまでは受付中。

### 3.3 ターム削除

| 条件 | エラーコード | メッセージ |
|------|-----------|---------|
| ステータスが `draft` 以外 | CONFLICT | draft ステータスのタームのみ削除可能です |

### 3.4 ステータス変更

| 条件 | エラーコード | メッセージ |
|------|-----------|---------|
| status API の責務外の遷移を指定 | INVALID_TRANSITION | このステータス変更は status API では許可されていません |
| `collecting` → `generating` を status API で試行 | INVALID_TRANSITION | generating への遷移はシフト生成APIから実行してください |
| `collecting` への遷移時に `request_deadline` 未設定 | VALIDATION_ERROR | 希望休受付を開始するには締切日を設定してください |
| `confirmed` への遷移時にフェーズ1必須制約違反が残っている | PHASE1_VIOLATION_REMAINING | 必須制約違反が解消されるまで確定できません |
| `generating` → `collecting` の強制ロールバックで5分未経過 | GENERATION_IN_PROGRESS | シフト生成が実行中です。5分以上経過後に再試行してください |

> **補足:** `generating` を含む遷移は generate API の副作用としてのみ発生する。status API は `draft` → `collecting`、`adjusting` → `confirmed`、`confirmed` → `adjusting` のような手動遷移のみを扱う。

### 3.5 年度一括登録

| 条件 | エラーコード | メッセージ |
|------|-----------|---------|
| year_start_date 未指定 | REQUIRED | 年度開始日を入力してください |
| 一部タームが既存と重複 | CONFLICT | 以下のタームが既存のタームと重複しています: [期間一覧] |

## 4. 希望休申請

### 4.1 希望休登録

| 条件 | エラーコード | メッセージ |
|------|-----------|---------|
| タームが `collecting` でない（スタッフ操作時） | FORBIDDEN | このタームの希望休受付は終了しています |
| 締切日を過ぎている（スタッフ操作時） | FORBIDDEN | 申請受付は締切日を過ぎています |
| 同一ターム3件超過 | LIMIT_EXCEEDED | 希望休は1タームにつき最大3日までです |
| 日付がターム期間外 | OUT_OF_RANGE | 選択された日付はターム期間外です |
| 重複申請 | CONFLICT | この日付は既に申請済みです |
| 対象スタッフが無効化済み | FORBIDDEN | 無効化されたスタッフの希望休は申請できません |

### 4.2 希望休取消

| 条件 | エラーコード | メッセージ |
|------|-----------|---------|
| 締切日を過ぎている（スタッフ操作時） | FORBIDDEN | 締切後の取消はできません |
| 存在しない申請 | NOT_FOUND | 指定された希望休が見つかりません |

## 5. シフト管理

### 5.1 シフト自動生成

| 条件 | エラーコード | メッセージ |
|------|-----------|---------|
| 有効スタッフ数 < `ceil((min_day_staff + 6) × 28 / (20 - 祝日数))` | INSUFFICIENT_STAFF | スタッフ数が不足しています（必要: N名以上、現在: M名）。祝日数を考慮した計算結果を表示 |
| 夜勤可能スタッフ数 < `ceil(6 × 28 / (20 - 祝日数))` | INSUFFICIENT_NIGHT_STAFF | 夜勤可能なスタッフが不足しています（必要: N名以上、現在: M名） |
| チームAから夜勤可能者 < 3名 | INSUFFICIENT_TEAM_STAFF | チームAに夜勤可能なスタッフが不足しています（必要: 3名以上） |
| チームBから夜勤可能者 < 3名 | INSUFFICIENT_TEAM_STAFF | チームBに夜勤可能なスタッフが不足しています（必要: 3名以上） |
| 生成エンジンの例外エラー | GENERATION_ERROR | シフト生成中にエラーが発生しました。ターム状態を受付中に戻しました |
| ターム状態が不正 | INVALID_STATE | この状態ではシフト生成を実行できません |
| 他のリクエストが生成中 | GENERATION_IN_PROGRESS | シフト生成が既に実行中です |

> **補足:** 55秒到達はエラーではなく成功レスポンスとして扱い、`meta.timed_out = true` を返す。

### 5.2 シフト保存

| 条件 | エラーコード | メッセージ |
|------|-----------|---------|
| `lock_version` 不一致（他セッションが先に保存） | OPTIMISTIC_LOCK_ERROR | 他の操作によりシフトデータが更新されました。最新データを読み込んでください |
| ターム状態が `adjusting` でない | INVALID_STATE | この状態ではシフトを保存できません |
| `lock_version` 未送信 | VALIDATION_ERROR | lock_version が必須です |

### 5.3 シフト手動編集（クライアント側バリデーション）

> **注意:** 編集・保存方式の正本は `01_機能要件定義書.md` と `05_API設計書.md` を参照。MVPでは単発編集APIや即時バッチ更新APIは提供せず、変更は「保存」ボタン押下時に `POST /api/terms/:termId/shifts/save` で一括送信する。下記はクライアント側で実施するバリデーションの仕様。

| 条件 | UI表示 |
|------|-------|
| 不正な shift_type が選択された | ドロップダウンに許可値のみ表示するため発生しない |
| `adjusting` でないタームのシフトを編集 | 編集UIを非活性化（グレーアウト）する |

### 5.4 シフト確定

| 条件 | エラーコード | メッセージ（確認ダイアログ） |
|------|-----------|---------|
| フェーズ1必須制約違反あり | PHASE1_VIOLATION_REMAINING | 必須制約違反が解消されるまで確定できません |
| 警告のみあり | — | 警告が N件 あります。シフト表を確定しますか？確定後スタッフに公開されます。 |
| 違反なし | — | シフト表を確定しますか？確定後スタッフに公開されます。 |

### 5.5 PDF出力

| 条件 | エラーコード | メッセージ |
|------|-----------|---------|
| ターム状態が `draft` / `collecting` / `generating` | INVALID_STATE | この状態ではPDF出力できません |
| 生成エラー | INTERNAL_ERROR | PDFの生成に失敗しました |
