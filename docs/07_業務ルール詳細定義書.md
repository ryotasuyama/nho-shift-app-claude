# 業務ルール詳細定義書

## NHO 看護師シフト管理アプリ — MVP

---

## 1. 勤務区分の正規定義

| 値 | 表示記号 | 業務意味 | 勤務時間帯 | 実働時間 |
|----|:---:|------|----------|:---:|
| `day` | 日 | 日勤 | 8:30〜17:15 | 7.75h |
| `evening` | 準 | 準夜勤 | 16:30〜翌1:15 | 7.75h |
| `night` | 深 | 深夜勤 | 0:30〜9:15 | 7.75h |
| `off` | 休 | 週休（自動配置） | — | 0h |
| `requested_off` | 希 | 希望休（スタッフ申請による休み） | — | 0h |
| `holiday_off` | 代 | 祝日代休（祝日勤務の代償休日） | — | 0h |

### 1.1 画面表示上の「祝」について

- 画面上の「祝」マークは **祝日マスタに登録された日付** に対して表示する背景装飾（薄桃背景）であり、`shift_type` の値ではない。
- 祝日に勤務する場合、`shift_type` は通常通り `day` / `evening` / `night` となる。
- 祝日に休む場合で、代休として付与された休みは `holiday_off` とする。

## 2. 休み区分のカウントルール

### 2.1 週休（H5）のカウント対象

| 勤務区分 | H5 週休としてカウント | 備考 |
|---------|:---:|------|
| `off` | **含む** | 自動配置の週休 |
| `requested_off` | **含む** | 希望休は週休の一部として扱う |
| `holiday_off` | **含まない** | 代休は週休8回に含めず、別枠 |

H5 の判定: `off` + `requested_off` の合計 = 8回/ターム

### 2.2 各制約での「休み」判定対象

| 制約 | `off` | `requested_off` | `holiday_off` | 判定説明 |
|------|:---:|:---:|:---:|------|
| H5（週休8回） | ○ | ○ | × | 週休カウント |
| H6（週休間隔最大5日） | ○ | ○ | △（許容） | `holiday_off` が間に挟まる場合は間隔超過を許容 |
| H7（土日連休1回） | ○ | ○ | ○ | いずれの休み区分でも連休として認定 |
| H8（平日連休1回） | ○ | ○ | ○ | 同上 |
| H9（48時間以上休み） | ○ | ○ | ○ | 前日が `evening` でないことが条件 |
| H10（法定週休4回） | ○ | ○ | ○ | 前日が `evening` でない24時間の完全な休み |

## 3. 連休・法定週休の厳密判定

### 3.1 連休の定義

- **連休**: 2日以上連続して休み（`off` / `requested_off` / `holiday_off` のいずれか）であること。
- **土日連休（H7）**: 土曜日と日曜日が両方とも休みであること。
- **平日連休（H8）**: 土日以外の2日以上連続する休み。

### 3.2 「48時間以上の休み」の判定（H9）

以下の **両方** を満たす連休を「48時間以上の休み」とする:

1. 2日以上連続の休み（`off` / `requested_off` / `holiday_off`）
2. 連休初日の前日の勤務が `evening`（準夜勤）**ではない**

> 準夜勤は翌1:15まで勤務があるため、翌日が休みでも実質的に24時間未満の休息となる。

### 3.3 「法定週休」の判定（H10）

以下を満たす休みを「法定週休」（準夜明けではない24時間の完全な休み）とする:

1. 当日が休み（`off` / `requested_off` / `holiday_off`）
2. 前日の勤務が `evening`（準夜勤）**ではない**

## 4. 祝日勤務と代休のルール

### 4.1 祝日勤務の対象

祝日マスタに登録された日に勤務する **全スタッフ** が代休付与の対象となる（日勤・準夜勤・深夜勤を問わない）。

### 4.2 代休の付与

- 祝日に勤務したスタッフ1名につき代休（`holiday_off`）を **1日** 付与する。
- 代休は **同一ターム内** で消化することを原則とする。

### 4.3 代休が消化できない場合

- シフト自動生成時に同一ターム内で代休配置が困難な場合、ソフト制約違反として警告を表示する（生成は継続）。
- 管理者は手動調整で代休の位置を変更可能。
- MVP では次タームへの繰越は対応しない。

### 4.4 代休の自動配置タイミング

シフト自動生成エンジンの処理順序:
1. 希望休の確定配置
2. 夜勤配置
3. 週休配置
4. **祝日勤務者の確認 → 代休配置**（週休とは別枠。日勤・準夜勤・深夜勤の全勤務者を対象に判定）
5. 日勤の残枠配置

## 5. 夜勤制約の詳細判定

### 5.1 夜勤ブロックの定義

- **夜勤ブロック**: 連続する夜勤（`evening` または `night`）の塊。
- 例: `night → evening` は1つの夜勤ブロック（2日）。

### 5.2 各制約の判定パターン

| 制約 | 禁止パターン（D=day, E=evening, N=night, O=off系） | 判定 |
|------|------|------|
| H11 | `E/N → E/N → E/N` | 夜勤が3日以上連続 → 違反 |
| H12 | `N → N` | 深夜勤が2日連続 → 違反 |
| H13 | `E → N` | 準夜の翌日が深夜 → 違反 |
| H14 | `E → O → N` | 準夜→休み→深夜 → 違反 |
| H15 | `E → O → D → N` | 準夜→休み→日勤→深夜 → 違反 |
| H16 | 夜勤ブロック間が2日以下 | 夜勤ブロック終了から次の夜勤ブロック開始まで3日以上空ける |

> **H14, H15 の「休み」**: `off` / `requested_off` / `holiday_off` のいずれも該当する。

### 5.3 ターム境界での判定（H19）

- 前タームが `confirmed` の場合、前タームの **最終3日間** のシフトデータを読み込み、H11〜H16 の判定に使用する。
- 前タームが `confirmed` **以外** の場合、ターム境界チェックはスキップし、管理者に警告を表示する。

## 6. 勤務時間の計算ルール

### 6.1 各勤務区分の実働時間

| 勤務区分 | 拘束時間 | 休憩 | 実働時間 |
|---------|---------|:---:|:---:|
| `day` | 8:45 | 1:00 | **7.75h** |
| `evening` | 8:45 | 1:00 | **7.75h** |
| `night` | 8:45 | 1:00 | **7.75h** |
| `off` / `requested_off` / `holiday_off` | — | — | **0h** |

### 6.2 ターム標準勤務時間

- 1ターム = 28日
- 週休8回 → 勤務日数 = 28 - 8 = 20日（`holiday_off` は別枠のため、代休がある場合は勤務日数が減る）
- 標準勤務時間 = 7.75h × 20日 = **155時間**（代休なしの場合）
- 代休がある場合: 155h - (代休日数 × 7.75h)

### 6.3 統計表示

- 各スタッフの合計勤務時間と標準勤務時間との差分を表示する。
- 代休を考慮した個別の標準時間を基準とする。

## 7. スタッフ削除・無効化のルール

### 7.1 論理削除の影響範囲

- 論理削除されたスタッフ（`is_active = false`）は:
  - **未確定のターム**（`draft` / `collecting` / `generating` / `adjusting`）のシフトから自動除外される。
  - **確定済みのターム**（`confirmed`）のシフトデータは保持される。
  - 希望休申請はブロックされる。
  - スタッフ一覧では「無効化済み」として別タブ表示。

### 7.1a 「自動除外」のDB上の意味

- 未確定タームでは、current working copy である `shift_entries` に無効化済みスタッフの行を残さない。
- スタッフを無効化した時点で、未確定タームの `shift_entries` にある当該 `staff_id` の行は削除対象とする。
- `POST /api/terms/:termId/shifts/generate`、`POST /api/terms/:termId/shifts/save`、`POST /api/terms/:termId/snapshots/:id/restore` は、いずれも `is_active = true` のスタッフのみを書き戻す。
- 既に `confirmed` のタームは履歴として扱うため、無効化後も既存の `shift_entries` を保持する。

### 7.2 UI上の文言ルール

| 操作 | 対象 | UI上の文言 |
|------|------|---------|
| スタッフ無効化 | staffs.is_active = false | 「無効化」（論理削除） |
| スタッフ復元 | staffs.is_active = true | 「復元」 |
| ターム削除 | terms の物理削除 | 「削除」（draft のみ可能） |

## 8. 仮パスワード生成ルール

管理者がスタッフを新規作成する際、またはパスワードを再発行する際に仮パスワードを生成する。

| 項目 | ルール |
|------|------|
| 文字列長 | 12文字 |
| 文字種 | 英大文字 + 英小文字 + 数字（記号は含めない） |
| 生成方式 | `crypto.randomBytes()` を使用したセキュアな乱数生成 |
| ポリシー適合 | パスワードポリシー（英数字混在8文字以上）を満たすことを保証する |
| 表示 | 生成後にレスポンスで1度だけ返却。サーバーには Supabase Auth がハッシュ化して保存 |

## 9. 希望休締切の判定方式

| 項目 | 内容 |
|------|------|
| 判定タイミング | API リクエスト時にリアルタイム判定 |
| 判定ロジック | `現在のJST日付 > request_deadline` の場合は締切後と判定 |
| バッチ処理 | 行わない。タームステータスは `collecting` のまま変更しない |
| スタッフ | 締切後は申請・変更・取消をAPIレベルでブロック（403 FORBIDDEN） |
| 管理者 | 締切後も代理入力・修正が可能 |

> **補足:** `request_deadline` は DATE として保持し、JST基準で締切日当日の23:59:59まで申請可能とする。
