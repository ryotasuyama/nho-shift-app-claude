## 2. テーブル定義

### 2.1 users — ユーザー（業務属性のみ）

> **方針:** 認証委任の正本は `01_機能要件定義書.md` と `06_技術スタック・アーキテクチャ設計書.md` を参照。本書ではデータ上の扱いのみ定義し、`public.users` には業務属性のみを保持する。`id` は Supabase Auth の `auth.users.id` と一致させる。

| カラム名 | データ型 | NULL | デフォルト | 説明 |
|---------|---------|:----:|----------|------|
| id | UUID | NO | | 主キー（Supabase Auth の `auth.users.id` と同一） |
| email | VARCHAR(255) | NO | | メールアドレス（表示・検索用。認証は Supabase Auth 側） |
| role | VARCHAR(20) | NO | 'staff' | ロール。`admin` / `staff`（`user_metadata.role` と同期） |
| must_change_password | BOOLEAN | NO | true | 初回パスワード変更フラグ。Supabase Auth の `user_metadata.must_change_password` と同期する |
| created_at | TIMESTAMP | NO | now() | 作成日時 |
| updated_at | TIMESTAMP | NO | now() | 更新日時 |

**インデックス:**
- `uk_users_email` — UNIQUE (email)

> **削除済みカラム（Supabase Auth に委任）:** `password_hash`、`failed_login_count`、`locked_until` はアプリテーブルから除外。Supabase Auth が内部で管理する。

> **`must_change_password` の Middleware 参照方針:** Next.js Middleware で全ページアクセス時にこのフラグを確認する必要がある。毎回 DB を参照するとパフォーマンスに影響するため、`must_change_password` の変更時に Supabase Auth の `user_metadata` にも同期する（`supabaseAdmin.auth.admin.updateUserById(id, { user_metadata: { must_change_password: value } })`）。Middleware では JWT 内の `user_metadata.must_change_password` を参照し、DB アクセスを回避する。

### 2.2 staffs — スタッフ

> **設計方針:** MVP では管理者（師長）も含め、全ユーザーが `staffs` レコードを持つ（`users` : `staffs` = 1:1 を全ロールに適用）。`user_id` は NOT NULL を維持する。純管理者専用アカウント（`staffs` レコードなし）は MVP スコープ外（`01_機能要件定義書.md §10.1` 参照）。

| カラム名 | データ型 | NULL | デフォルト | 説明 |
|---------|---------|:----:|----------|------|
| id | UUID | NO | gen_random_uuid() | 主キー |
| user_id | UUID | NO | | 外部キー → users.id |
| staff_code | VARCHAR(10) | NO | | 業務用スタッフコード（例: `N001`）。一意制約 |
| name | VARCHAR(60) | NO | | 氏名 |
| experience_years | INTEGER | NO | | 経験年数（1〜50） |
| team | VARCHAR(1) | NO | | 所属チーム。`A` / `B` |
| night_shift_available | BOOLEAN | NO | true | 夜勤可否 |
| is_active | BOOLEAN | NO | true | 有効フラグ（論理削除用） |
| created_at | TIMESTAMP | NO | now() | 作成日時 |
| updated_at | TIMESTAMP | NO | now() | 更新日時 |

**インデックス:**
- `uk_staffs_user_id` — UNIQUE (user_id)
- `uk_staffs_staff_code` — UNIQUE (staff_code)
- `idx_staffs_team` — (team)
- `idx_staffs_active` — (is_active)

### 2.3 terms — ターム

| カラム名 | データ型 | NULL | デフォルト | 説明 |
|---------|---------|:----:|----------|------|
| id | UUID | NO | gen_random_uuid() | 主キー |
| start_date | DATE | NO | | ターム開始日 |
| end_date | DATE | NO | | ターム終了日（start_date + 27日） |
| fiscal_year | INTEGER | NO | | 年度（開始日から自動導出。4月〜翌3月。例: 2026年4月→2026、2027年2月→2026） |
| status | VARCHAR(20) | NO | 'draft' | ステータス。`draft` / `collecting` / `generating` / `adjusting` / `confirmed` |
| generating_started_at | TIMESTAMP | YES | | `generating` 遷移日時。スタック復旧の経過時間判定に使用する。`generating` 以外のステータスでは NULL |
| request_deadline | DATE | YES | | 希望休申請締切日。JST基準で当日23:59:59まで有効な締切日として扱う |
| min_day_staff | INTEGER | NO | 7 | 日勤最低人数（管理者設定可能） |
| lock_version | INTEGER | NO | 1 | 楽観的ロック用バージョン番号。保存成功時に +1 インクリメントする |
| created_at | TIMESTAMP | NO | now() | 作成日時 |
| updated_at | TIMESTAMP | NO | now() | 更新日時 |

**インデックス:**
- `idx_terms_start_date` — (start_date)
- `idx_terms_status` — (status)
- `idx_terms_fiscal_year` — (fiscal_year)

**制約:**
- `chk_terms_dates` — CHECK (end_date = start_date + 27)
- `chk_terms_status` — CHECK (status IN ('draft','collecting','generating','adjusting','confirmed'))

**締切日の解釈:**
- `request_deadline` は日付のみ保持する
- スタッフの希望休申請可否は `現在のJST日付 > request_deadline` で判定する
- そのため締切日当日の 23:59:59 JST までは申請・取消が可能

**楽観的ロック:**
- シフトデータの保存時に専用の `lock_version`（整数）カラムを使用する
- `updated_at` は楽観的ロックには使用しない（別用途の更新でも競合扱いになる問題を避けるため）
- クライアントがシフトデータ取得時に受け取った `lock_version` を保存リクエストに含め、サーバー側で現在値と一致するかチェックする
- 一致した場合は保存後に `lock_version` を +1 インクリメントする

**削除ルール:**
- `draft` ステータスのタームのみ物理削除可能（CASCADE で関連する shift_requests も削除）
- `draft` 以外のタームはアプリケーションレベルで削除を禁止する

### 2.6 shift_snapshots — シフト編集履歴

| カラム名 | データ型 | NULL | デフォルト | 説明 |
|---------|---------|:----:|----------|------|
| id | UUID | NO | gen_random_uuid() | 主キー |
| term_id | UUID | NO | | 外部キー → terms.id |
| version | INTEGER | NO | | 世代番号（1〜5） |
| data | JSONB | NO | | シフト全体のスナップショット |
| created_by | UUID | NO | | 外部キー → users.id（変更者） |
| created_at | TIMESTAMP | NO | now() | 作成日時 |

**インデックス:**
- `uk_shift_snapshots` — UNIQUE (term_id, version)

**バージョン番号の管理ルール:**
- version は連番でインクリメントする（1, 2, 3, ...）。上限5には固定しない
- 6世代目の作成時、最も小さい version のレコードを削除し、常に最新5世代を保持する
- 例: version 1〜5 が存在 → 新規保存で version 6 を作成し version 1 を削除 → 保持: 2〜6

**data カラムの構造例:**
```json
{
  "entries": [
    {
      "staff_id": "uuid",
      "date": "2026-04-01",
      "shift_type": "day",
      "is_manual_edit": true
    }
  ],
  "violations": [
    {
      "constraint_id": "H11",
      "staff_id": "uuid",
      "message": "夜勤の連続が3日以上"
    }
  ]
}
```

- `data.entries` は復元対象の正本であり、`shift_type` と `is_manual_edit` をそのまま保持する
- snapshot 復元時は `data.entries` を current working copy に書き戻し、`violations` や統計値は復元後の `shift_entries` から再計算する

### 2.8 audit_logs — 操作ログ

> **方針:** 非機能要件で定義された操作ログ（365日保持）を永続化するテーブル。シフト編集・マスタ変更等の重要操作を記録する。ログに個人情報（氏名・メールアドレス）を含めず、ユーザーIDのみを記録する。

| カラム名 | データ型 | NULL | デフォルト | 説明 |
|---------|---------|:----:|----------|------|
| id | UUID | NO | gen_random_uuid() | 主キー |
| user_id | UUID | NO | | 操作者のユーザーID（users.id） |
| action | VARCHAR(50) | NO | | 操作種別（後述の列挙値） |
| resource_type | VARCHAR(30) | NO | | 対象リソース種別（`staff` / `term` / `shift` / `request` / `snapshot` / `holiday`） |
| resource_id | UUID | YES | | 対象リソースのID（特定リソースに紐付かない場合はNULL） |
| detail | JSONB | YES | | 操作の詳細情報（変更前後の値など） |
| created_at | TIMESTAMP | NO | now() | 操作日時 |

**インデックス:**
- `idx_audit_logs_user_id` — (user_id)
- `idx_audit_logs_action` — (action)
- `idx_audit_logs_created_at` — (created_at)
- `idx_audit_logs_resource` — (resource_type, resource_id)

**action 値の定義:**

| 値 | 説明 |
|----|------|
| `staff.create` | スタッフ新規作成 |
| `staff.update` | スタッフ情報更新 |
| `staff.deactivate` | スタッフ無効化 |
| `staff.restore` | スタッフ復元 |
| `term.create` | ターム作成 |
| `term.update` | ターム更新 |
| `term.delete` | ターム削除 |
| `term.status_change` | タームステータス変更 |
| `shift.generate` | シフト自動生成 |
| `shift.edit` | シフト手動編集 |
| `shift.save` | シフト保存 |
| `shift.confirm` | シフト確定 |
| `shift.revert` | シフト差し戻し |
| `snapshot.restore` | スナップショット復元 |
| `request.create` | 希望休申請 |
| `request.delete` | 希望休取消 |
| `auth.password_reset` | パスワード再発行 |

**detail カラムの構造例:**
```json
{
  "term_id": "uuid",
  "staff_id": "uuid",
  "changes": [
    { "date": "2026-04-06", "from": "day", "to": "evening" }
  ]
}
```

**保持期間:** 365日。365日を超えたレコードは定期バッチで削除する。

**定期削除バッチ:**
- エンドポイント: `GET /api/cron/cleanup-audit-logs`（認証不要だが `CRON_SECRET` ヘッダーで保護）
- スケジュール: 毎日 3:00 JST（Vercel Cron で設定）
- 処理: `DELETE FROM audit_logs WHERE created_at < NOW() - INTERVAL '365 days'`（1回あたり最大1000件削除。残件があれば翌日に継続）
- vercel.json に Cron 設定を追加する
